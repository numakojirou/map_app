# roadmap — 現状の制約と今後の拡張候補

「社内メンバーの現場配置を地図で共有する」という本来用途に対して、現状はまだ **静的なサンプル地点を表示するだけ** の段階。実用化に向けて拡張すべき領域を整理する。

## 現状の主な制約

| # | 制約 | 影響 |
|---|---|---|
| C1 | マーカーが静的 JSON | 配置を変えるたびに JSON 編集 + 再デプロイが必要 |
| C2 | 認証がない | URL を知っていれば誰でも見られる |
| C5 | 検索／フィルタが無い | メンバー数や現場数が増えると探しづらい |
| C6 | テストが実質ゼロ | リファクタの安全網が無い（`App.test.js` は CRA 雛形のまま落ちる） |
| C7 | UI が地図のみ | 一覧／凡例・カテゴリ表示などの補助情報が無い |

> ※ 旧 C3（メンバーと地点の区別）と C4（更新時刻なし）は Phase 2 で解消済み。

### Phase 3 — 編集と永続化 ✅ 完了

- [x] **マーカー編集 UI（3-A）** — 地図クリック / ドラッグ / 編集 / 削除
  - ヘッダー「+ メンバー追加」→ 追加モード（cursor crosshair）→ 地図クリックでフォーム
  - 既存マーカークリック → popup に「編集」「削除」ボタン
  - マーカードラッグ → ドロップ時に `window.confirm` で確認、OK で位置 +
    `updatedAt` を保存、キャンセルで元位置に戻る（誤操作防止：マーカーの移動
    は人事異動など明確な配置変更時のみという運用想定のため）

  - 操作結果はトーストで通知（2.4 秒で消える）
  - 新規 ID は `nextMemberId` で連番採番（`m-007`, `m-008`...）
  - SeedPanel に確認ダイアログを追加（誤投入ガード）
- [x] **永続化バックエンド（3-B）** — Firebase Firestore へ移行
  - リージョン: `asia-northeast1`（東京）、データベース: `(default)`
  - コレクション `members` に 1 メンバー = 1 ドキュメント（ID は `m-001` 形式）
  - `src/data/membersRepo.js` が CRUD + リアルタイム購読を提供
  - 初期データ投入は `SeedPanel`（Firestore が空のときだけ表示される一回限りの UI）
  - `firestore.rules` でサーバ側にも allowlist を持つ defense-in-depth
- [x] **認証（3-C）** — Firebase Authentication / Google サインイン
  - 現状: クライアントサイドの allowlist（`k.kakinuma0001@gmail.com`）で検証中
  - 将来: 会社の Google Workspace ドメインが決まったら `src/auth/accessControl.js`
    と `firestore.rules` の両方に追加して切り替え
  - 未ログイン → `LoginPage`、ログイン済 allowlist 外 → `AccessDenied`、許可済 → `MapView`

## 技術的に検討が要るポイント

- **SeedPanel の意図せぬ再投入リスク** — Phase 3-A で `window.confirm` のガードを
  入れたが、根本対策としては「初回投入済フラグを別ドキュメントに保存」「dev
  builds 限定で表示」などがより堅い
- **新規メンバー ID の同時採番衝突** — 1〜2 人運用前提だが、複数人が同時に
  「+ メンバー追加」したら同じ `m-007` を取り合う。Firestore transaction で
  カウンタドキュメントを atomic に増やす方式に置換可
- **react-leaflet 5 + React 19 の相性** — 動作はしているが、StrictMode 二重描画で
  Leaflet が文句を言うケースが将来出るかもしれない（要観察）
- **createMarkerIcon が毎レンダー新インスタンス** — 6 件規模なら問題なし。数百件に
  なるなら `useMemo` キャッシュを検討
