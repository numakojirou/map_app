# roadmap — 現状の制約と今後の拡張候補

「社内メンバーの現場配置を地図で共有する」という本来用途に対して、現状はまだ **静的なサンプル地点を表示するだけ** の段階。実用化に向けて拡張すべき領域を整理する。

## 現状の主な制約

| # | 制約 | 影響 |
|---|---|---|
| C1 | マーカーが静的 JSON | 配置を変えるたびに JSON 編集 + 再デプロイが必要 |
| C2 | 認証がない | URL を知っていれば誰でも見られる |
| C3 | メンバー単位の概念がない | `name` がメンバー名なのか地点名なのか区別できない |
| C4 | 更新時刻が無い | 情報の鮮度が分からない |
| C5 | 検索／フィルタが無い | メンバー数や現場数が増えると探しづらい |
| C6 | テストが実質ゼロ | リファクタの安全網が無い（`App.test.js` は CRA 雛形のまま落ちる） |
| C7 | UI が地図のみ | 一覧／凡例／カテゴリ表示などの補助情報が無い |

## 拡張アイデア（優先度の目安付き）

### Phase 1 — 基盤整備（小さく始める）

- [ ] **README をこのアプリ用に書き直す** — 現状は CRA 雛形のまま
- [ ] **App.test.js を実態に合わせるか削除** — `learn react` を探すテストは無意味
- [ ] **未使用ファイル整理** — `App.css` / `logo.svg` 削除
- [ ] **データ型に `id` と `updatedAt` を追加** — React `key` の安定化と鮮度表示

### Phase 2 — 用途への接続（メンバー × 現場）

- [ ] **データモデルを「メンバー」と「現場」に分離**
  - メンバー: id / 名前 / 役職 / アバター
  - 配置: メンバー id × 現場 (lat/lng) × updatedAt
- [ ] **ポップアップ表示の刷新** — 「誰が」「どこに」「いつ更新」を表示
- [ ] **カテゴリでアイコン色を変える** — 一目で区別したい

### Phase 3 — 編集と永続化

- [ ] **マーカー編集 UI** — 地図クリック／ドラッグでメンバーを配置
- [ ] **永続化バックエンド**
  - 案 A: **Firebase Firestore** — Hosting と同じプロジェクトに同居。リアルタイム同期も得意
  - 案 B: 小さな静的 API（Cloud Functions など）+ JSON
- [ ] **認証** — Firebase Authentication（社内ドメイン縛り or 招待制）

### Phase 4 — 体験向上

- [ ] **メンバー一覧／検索** — 名前・現場・カテゴリで絞り込み
- [ ] **凡例（カテゴリ・更新時間）**
- [ ] **モバイル最適化** — 現場から自分の位置を更新するユースケース
- [ ] **地図中心・初期 zoom の自動調整** — 全マーカーが収まるよう fit bounds

### Phase 5 — 運用

- [ ] **CI/CD** — GitHub Actions で `main` push → Firebase Hosting 自動デプロイ
- [ ] **環境分離** — `staging` / `production` を Firebase プロジェクトレベルで分ける
- [ ] **テスト整備** — マーカー描画のスナップショット、データ読み込みの mock テスト

## 技術的に検討が要るポイント

- **Firestore を採用すると `markers.json` のフェッチは廃止** → `App.js` のデータ取得層を抽象化しておくと差し替えが楽
- **react-leaflet 5 + React 19 の相性** — 動作はしているが、StrictMode 二重描画で Leaflet が文句を言うケースが将来出るかもしれない（要観察）
- **マーカー数が増えた場合のパフォーマンス** — 100 件超なら `react-leaflet-cluster` などのクラスタリング検討
