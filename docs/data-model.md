# data-model — マーカーデータの形式と運用

## ファイル位置

`public/markers.json`

CRA は `public/` 配下を静的配信するため、ブラウザからは `GET /markers.json` でそのまま取得できる。`App.js` の `useEffect` 内 `fetch("/markers.json")` がここを叩いている。

## スキーマ

JSON 配列。各要素は社内メンバー 1 人を表す：

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | string | ✅ | 安定 ID（React の `key` に使用、例: `"m-001"`） |
| `name` | string | ✅ | メンバー名（例: `"山田 太郎"`） |
| `site` | string | ✅ | 配置先の現場・拠点名（例: `"新宿支店"`、`"在宅勤務（大田区）"`） |
| `lat` | number | ✅ | マーカーを立てる緯度 |
| `lng` | number | ✅ | マーカーを立てる経度 |
| `category` | string | ✅ | 働き方区分。`"出社"` / `"在宅"` / `"ハイブリッド"` のいずれか |
| `updatedAt` | string (ISO 8601) | ✅ | 最終更新日時（例: `"2026-04-15T09:00"`） |

### サンプル（現状の中身そのまま）

```jsonc
[
  {
    "id": "m-001",
    "name": "山田 太郎",
    "site": "本社オフィス",
    "lat": 35.681236, "lng": 139.767125,
    "category": "出社",
    "updatedAt": "2026-04-15T09:00"
  },
  {
    "id": "m-002",
    "name": "佐藤 花子",
    "site": "新宿支店",
    "lat": 35.690921, "lng": 139.700258,
    "category": "ハイブリッド",
    "updatedAt": "2026-04-22T10:30"
  }
  // ... 計 6 名（架空メンバー）
]
```

## 表示への反映

`App.js` で配列を回し、各要素を 1 つの `<Marker key={member.id}>` として描画。`<Popup>` の構成は次の 3 行：

```
<strong>{name}</strong>
現場: {site}
最終更新: 2026/04/15 09:00   ← formatUpdatedAt で整形
```

`category` は **データとしては保持しているが、現状ポップアップには表示していない**（用途上、種別表示は不要との判断）。将来アイコン色分けや一覧フィルタを足すときの分類軸として使う想定。

## 運用フロー（現状）

1. `public/markers.json` を直接編集（メンバーの現場移動が発生したタイミング）
2. `npm start` で目視確認
3. `npm run build` → `firebase deploy --only hosting`

> 日次運用ではなく、**メンバーの現場移動契機で編集される**運用を想定。`updatedAt` は数日〜数週間前の値が入っているのが普通。

→ 編集 UI が無いため、毎回 JSON 編集 + 再ビルド + 再デプロイが必要。これは `roadmap.md` の Phase 3（編集 UI と永続化）で解消する想定。

## 今後のスキーマ拡張候補

- `role` … 役職（部長・課長など）。一覧表示や絞り込みに使う
- `phone` / `slack` … 連絡先。ポップアップから即連絡できるように
- `avatar` … 画像 URL。ポップアップに顔写真を出す
- `notes` … 自由記入欄（特記事項）

これらを追加する場合、`App.js` のポップアップ描画と、永続化先（Firestore へ移行する場合）を合わせて変更する必要がある。
